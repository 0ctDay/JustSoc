# JustSoc deployment package

This directory is a self-contained deployment package for the JustSoc probe stack
(probe + threat-engine + dispatcher/monitor). All paths the install script
references live under this directory, so a probe host only needs to copy
`deploy/` onto the box.

## Layout

```
deploy/
├── bin/                              # binaries (produced by ./build.sh)
│   ├── selk-probe-linux-amd64
│   └── threat-engine
├── monitor/                          # python monitor + control helper (copied from ../monitor/)
│   ├── selk-runtime-monitor.py
│   └── selk-runtime-control.sh
├── suricata-rules/                   # *.rules consumed by suricata (copied from ../suricata-rules/)
├── configs/
│   ├── probe-stack.env.example       # shipped, never edited in place
│   ├── probe-stack.env               # the ONLY file humans edit (created from .example)
│   ├── probe.example.yaml            # probe init template (shipped)
│   ├── engine-rules.yaml             # threat-engine rule data (shipped)
│   ├── probe.yaml                    # generated by init-config.sh
│   ├── suricata.generated.yaml       # generated by init-config.sh
│   ├── probe-whitelist.generated.bpf # generated by init-config.sh
│   ├── engine.conf                   # generated by install.sh
│   ├── runtime-monitor.env           # generated by install.sh
│   └── dispatcher.shared_secret      # generated by install.sh (hmac mode)
├── build.sh / build.ps1              # build binaries + sync static assets
├── install-suricata.sh              # install Suricata if missing (called by init-config.sh)
├── init-config.sh                    # generate host-specific probe configs
├── generate-secret.sh                # create / rotate / show the HMAC dispatcher secret
├── install.sh                        # write generated configs + systemd units, start services
└── uninstall.sh                      # stop and remove systemd units
```

## Unified flow

```bash
# 1. Build (Linux or Windows host with Go toolchain):
./build.sh                            # or .\build.ps1 on Windows

# 2. Init in one run. On a TTY this is interactive: it creates
#    configs/probe-stack.env from .example, prompts for the two critical params
#    (capture interface picked from a list + Kafka brokers/user/password), then
#    generates probe.yaml + suricata.generated.yaml -- no second run, no manual
#    edit. It also auto-installs Suricata if missing (needs root/sudo), since the
#    generated config is derived from the host's /etc/suricata/suricata.yaml.
#      SELK_NONINTERACTIVE=1  create the env and exit (edit by hand, then rerun)
#      SELK_RECONFIGURE=1     re-prompt interface/Kafka even if the env exists
#      SELK_SKIP_SURICATA_INSTALL=1  skip Suricata install (managed elsewhere)
#    Tune anything else (sasl mechanism, topics, ...) by editing probe-stack.env.
./init-config.sh

# 3. Install systemd units and start services. On first run this also
#    generates configs/dispatcher.shared_secret (HMAC mode) and prints the
#    secret to the console -- copy it to the platform / client side.
sudo ./install.sh

# Restart helper (uses configs/dispatcher.shared_secret):
./monitor/selk-runtime-control.sh restart probe
./monitor/selk-runtime-control.sh restart engine
./monitor/selk-runtime-control.sh restart all
```

## HMAC secret management

`install.sh` auto-runs `generate-secret.sh` the first time it sees that
`configs/dispatcher.shared_secret` is missing, printing the secret once.
To re-print or rotate later:

```bash
./generate-secret.sh --show    # print the current secret
./generate-secret.sh --force   # rotate: regenerate and print a new secret
                               # then run: sudo systemctl restart selk-probe-dispatcher
```

The secret is written 0600 to the path in `SELK_DISPATCH_SHARED_SECRET_FILE`
(default `configs/dispatcher.shared_secret`) and never committed to source.

## Config ownership

| File                                  | Owner             | When written              |
| ------------------------------------- | ----------------- | ------------------------- |
| `configs/probe-stack.env`             | human             | edited in place           |
| `configs/probe.example.yaml`          | shipped           | `./build.sh`              |
| `configs/engine-rules.yaml`           | shipped           | `./build.sh`              |
| `configs/probe.yaml`                  | generated         | `./init-config.sh`        |
| `configs/suricata.generated.yaml`     | generated         | `./init-config.sh`        |
| `configs/probe-whitelist.generated.bpf` | generated       | `./init-config.sh`        |
| `configs/engine.conf`                 | generated         | `./install.sh`            |
| `configs/runtime-monitor.env`         | generated         | `./install.sh`            |
| `configs/dispatcher.shared_secret`    | generated (hmac)  | `./install.sh`            |
| `suricata-rules/*.rules`              | shipped           | `./build.sh`              |
| `monitor/selk-runtime-monitor.py`     | shipped           | `./build.sh`              |
| `monitor/selk-runtime-control.sh`     | shipped           | `./build.sh`              |

Do not hand-edit generated files; edit `configs/probe-stack.env` and rerun the
appropriate step.

## Path conventions

- All paths inside generated configs are relative to `deploy/configs/`:
  - `rules.path: engine-rules.yaml` → `deploy/configs/engine-rules.yaml`
  - `rules.suricata_dir: ../suricata-rules` → `deploy/suricata-rules/`
  - `assets.path: ../assets/current/assets.yaml` → `deploy/assets/current/assets.yaml`
- The systemd units set `WorkingDirectory=` to `deploy/` so the probe's
  CWD-relative paths (`./suricata-rules`, `configs/...`) resolve correctly.

## Uninstall

```bash
sudo ./uninstall.sh                   # stop/remove services
sudo REMOVE_LOGS=1 ./uninstall.sh     # also wipe /var/log/selk/*.log
```
